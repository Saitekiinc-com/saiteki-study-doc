# AIネイティブ・エンジニアリング指針

AI（LLM）の進化は、ソフトウェアエンジニアリングのあり方を根本から変えようとしています。
本ドキュメントでは、エンジニアに求められる役割がどう変化しているのか（As-Is / To-Be）、そして今後どのように学習・開発を進めるべきかについて解説します。

## 1. As-Is / To-Be: エンジニア像の変化

AIの登場前後で、エンジニアに求められる価値は以下のようにシフトしています。

| 観点 | 🕰️ As-Is (Before AI) | 🚀 To-Be (After AI) |
| :--- | :--- | :--- |
| **役割** | **Code Writer**<br>(コードを書く人) | **Architect & Conductor**<br>(設計し、AIを指揮する人) |
| **価値の源泉** | **How (実装力)**<br>文法知識、タイピング速度、記憶力 | **What & Why (設計・判断力)**<br>要件定義、技術選定、品質保証 |
| **開発プロセス** | 人間が0から100まで書く | AIが下書きし、人間が**レビュー・修正**する |
| **ボトルネック** | コーディングの時間とスキル | **言語化能力** (やりたいことを正確に伝える力) |
| **必要な知識** | 特定の言語・フレームワークの深い知識 | 幅広い技術スタックの理解と、**原理原則** |

### 変化のポイント
これまでは「仕様通りに動くコードを書けること」が最大の価値でした。
これからは、**「AIという優秀だが指示待ちの部下を使いこなし、ビジネス価値を最大化すること」**が価値になります。

## 2. Why: なぜ変わるのか？

### 「How（実装）」のコモディティ化
AIは、既知のパターンであれば人間よりも圧倒的に速く、正確にコードを書くことができます。
「ReactでTodoアプリを作る」「SQLでデータを抽出する」といった**「How（どうやるか）」の価値は、限りなくゼロに近づいています。**

### 「What（要件）」と「Why（設計）」への価値移動
一方で、AIは「何を作るべきか（What）」や「なぜその技術を選ぶのか（Why）」を自律的に決めることは苦手です。
ビジネスの課題を解決するために**最適なシステムを定義し、AIの出力が正しいかを判断する**、上流（要件定義・設計）と下流（テスト・QA）の工程に価値が移動しています。

## 3. Future: 今後の展望

### フルスタック化の加速
AIの支援により、未知の言語やフレームワークを扱う障壁が劇的に下がりました。
フロントエンドエンジニアがバックエンドやインフラ（AWS/Terraform）を触る、あるいはその逆が当たり前になります。
**「専門外だからできない」という言い訳は通用しなくなり、サービス全体を俯瞰できるフルスタックな能力**が求められます。

### エンジニアは「指揮者」へ
エンジニアは、オーケストラの指揮者のような存在になります。
AI（演奏家）に指示を出し、全体の調和（アーキテクチャ）を取り、最終的なアウトプット（プロダクト）の品質に責任を持ちます。

### 職種別の変化 (AWS/Infra, QA)

| 職種 | 🕰️ As-Is (Before AI) | 🚀 To-Be (After AI) |
| :--- | :--- | :--- |
| **AWS / Infra** | **Configurator**<br>IaC (Terraform) を書く、設定値を管理する | **Platform Engineer**<br>スケーラブルな構成を設計する、コストとガバナンスを管理する |
| **QA / Tester** | **Test Executor**<br>テストケースを書き、手動で実行する | **Quality Strategist**<br>「何をテストすべきか」を戦略立て、AIによる自動テストを統制する |

*   **インフラ**: 「コンソールでの手作業」ではAIの出番は少ないですが、**IaC (Infrastructure as Code)** の領域では絶大な威力を発揮します。
    *   **活用例**: Terraform/CDKコードの生成、複雑なIAMポリシーの記述とセキュリティチェック、障害時のログ分析（AIOps）。
    *   これにより、人間は「設定作業」から解放され、「可用性・コスト・セキュリティを考慮したアーキテクチャ設計」に集中します。
*   **QA**: テストコードの作成や網羅的なケース出しはAIが代行します。人間は「ユーザーにとってのクリティカルな体験は何か」を定義し、**品質の基準**を作る役割になります。

## 4. Action: 今やるべきこと

この変化に対応するために、本研修では以下のスキル習得を重視します。

### ① 基礎知識と原理原則 (Foundation)
AIの出力が「正しいか」「安全か」を判断するためには、基礎知識が不可欠です。
*   **レビュー力**: AIのハルシネーション（嘘）を見抜く。
*   **設計力**: AIに適切な指示を出すための青写真を描く。

### ② 言語化能力 (Prompt Engineering)
曖昧な指示は、バグの温床になります。
*   **要件定義**: 「何がしたいか」を論理的かつ具体的に言葉にする。
*   **コンテキストエンジニアリング**: 前提条件や制約をAIに正しく伝える。

### ③ 多角的な検証能力 (Verification)
「動けばOK」ではありません。AIが見落としがちな非機能要件を人間が補完します。
*   **セキュリティ**: 脆弱性がないか。
*   **パフォーマンス**: 無駄な処理がないか。
*   **UX/アクセシビリティ**: ユーザーにとって使いやすいか。
