# メンタリングガイドライン

メンターの役割は、答えを教えることではなく、研修生が自ら答えにたどり着けるように導くことです。

## 1. メンターの心構え
*   **心理的安全性を確保する**: 「こんなこと聞いてもいいのかな？」と思わせない雰囲気作り。質問は歓迎する。
*   **プロセスを評価する**: 結果だけでなく、どのように考え、どのように解決しようとしたかを評価する。
*   **自走力を育てる**: すぐに答えを教えず、ヒントを出したり、調べ方を教えたりする。

## 2. 効果的なフィードバック
*   **具体的であること**: 「もっと良くして」ではなく、「ここの変数名は役割を表す名前に変更しましょう」と伝える。
*   **ポジティブから始める**: まずはできたことや努力を認め、その後に改善点を伝える（サンドイッチ法）。
*   **Whyを説明する**: なぜその修正が必要なのか、理由や背景（パフォーマンス、可読性、保守性など）を説明する。

## 3. 初心者が陥りやすい落とし穴と対処法

### ケース1: エラーメッセージを読まない
*   **対処法**: 一緒にエラーメッセージを読み、どこに何が書いてあるか（ファイル名、行数、エラー内容）を解説する。「エラーは友達」という意識を植え付ける。

### ケース2: 一度にすべてを解決しようとする
*   **対処法**: 問題を小さく分割するように促す。「まずはこのボタンが表示されるところまでやってみよう」など、小さな成功体験を積ませる。

### ケース3: コピペで動いたからヨシとする
*   **対処法**: 「なぜ動いたのか？」を説明してもらう。理解できていない場合は、コードの各行が何をしているか解説してもらう。

## 4. コードレビューのポイント
*   **ロジックの正しさ**: 仕様通りに動くか。
*   **可読性**: 変数名や関数名は適切か。複雑すぎるロジックはないか。
*   **型安全性**: `any` を多用していないか。適切な型定義がされているか。
*   **コンポーネント設計**: 再利用性や責務の分離は適切か。
